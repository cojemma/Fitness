要設計一個「完整」的訓練模板（Workout Template）系統，不能只把它當作一個「複製貼上」的工具。對於像你這樣關注重訓與增肌的開發者來說，模板的核心價值在於**「標準化訓練流程」與「追蹤漸進式超負荷（Progressive Overload）」**。

一個完整的模板功能應涵蓋以下四個維度：

1. 核心資料模型 (Data Schema)
模板不只是 Workout 的複製品，它是一套「預期目標」。在資料庫設計時，你需要考慮：

基礎資訊： 模板名稱（如：推系列 A）、目標肌群、預計總時長、備註說明。

動作組成 (Exercise Entry)：

預設組數 (Sets)： 預計要做幾組。

預設目標 (Targets)： 每組的預期次數 (Reps)、重量 (Weight) 或 RPE (自覺強度)。

組間休息時間 (Rest Timer)： 針對該動作設定特定的休息秒數。

層級關係： 確保 Template -> TemplateExercise -> TemplateSet 的關係清晰，這在 SQL (Room) 中通常需要三張表來達成。

2. 靈活的編輯功能 (Editing Logic)
用戶在制定計畫時，需要高度的自由度：

超級組 (Supersets) 支援： 能夠將兩個或多個動作群組化，提示用戶交替進行（這對提高訓練效率非常重要）。

動作替換 (Exercise Swapping)： 如果當天某器材被佔用，用戶應能在執行時臨時更換動作，並選擇是否「永久更新模板」。

排序調整： 簡單的拖拉排序，確保訓練順序符合生理需求（如：先做大重量多關節動作）。

3. 執行時的自動化 (Execution & UX)
當用戶點擊「開始訓練」並載入模板時，系統應具備以下智慧功能：

預載上次數據 (Last Session Data)： 自動顯示上次該模板的重量與次數（這就是追蹤漸進式超負荷的核心）。

自動跳轉計時器： 完成一組勾選後，根據模板設定的休息時間自動啟動倒數。

一鍵儲存為新模板： 如果用戶今天隨興練了一套很棒的流程，應能直接將「歷史記錄」存回「模板庫」。

4. SDK 層級的高級邏輯 (The Pro Features)
既然你正在開發 SDK，這幾點能讓你的 SDK 具備競爭力：

熱身組邏輯 (Warm-up Sets)： 模板應能標記哪些是熱身組，不計入訓練總容量（Volume）統計。

百分比重量 (Percentage-based Training)： 支援根據用戶的 1RM (最大重量) 設定目標（例如：今天的臥推是 1RM 的 70%），SDK 自動計算具體重量。

版本控制： 當模板修改後，不應影響舊的歷史記錄（歷史記錄應存儲當時的快照，而非關聯到會變動的模板）。
